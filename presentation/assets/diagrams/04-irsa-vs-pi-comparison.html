IRSA vs EKS Pod Identity — 5축 비교

같은 문제(Pod에 IAM 권한 부여)를 두 방식이 어떻게 다르게 푸는가

IRSA
IAM Roles for Service Accounts (2019~)
EKS Pod Identity
pods.eks.amazonaws.com (2023.11~)
1Trust 메커니즘
클러스터별 OIDC IdP 등록 필요
Principal.Federated = 클러스터 OIDC provider ARN
→ 신규 클러스터마다 IAM OIDC provider 생성
단일 service principal로 고정
Principal.Service = pods.eks.amazonaws.com
→ 모든 클러스터·계정이 같은 trust 재사용
2Role 연결 위치
두 곳 (IAM + K8s) 양쪽에 설정
IAM Role trust policy 의 sub/aud condition
+ ServiceAccount eks.amazonaws.com/role-arn annotation
한 곳 (Pod Identity Association)
EKS API CreatePodIdentityAssociation 한 번 호출
unique key = (cluster, namespace, serviceAccount)
3STS quota 사용
사용 Customer 계정 STS 호출
SDK가 Pod 안에서 직접 sts:AssumeRoleWithWebIdentity
→ Account×Region STS request quota 소비
미사용 EKS 서비스가 호출
Agent가 eks-auth:AssumeRoleForPodIdentity 호출
→ Customer STS quota 소비 없음
4멀티클러스터 trust 갱신
클러스터 추가마다 모든 Role의 trust 갱신
Trust policy 길이 한도 2,048 → 8,192자
→ Role 하나에 trust 관계 보통 ~4개, 증액해도 ~8개
Trust 변경 불필요 — Association만 추가
Blue/Green 클러스터 전환 시 IAM 변경 0건
새 클러스터에서 Association만 생성
5의존 컴포넌트
EKS OIDC IdP + Pod Identity Webhook
Webhook이 SA annotation 보고 env+projected token 주입
(EKS 내장, 별도 add-on 없음)
Pod Identity Agent DaemonSet (Add-on)
노드 link-local 169.254.170.23:80 + 노드 IAM Role 권한
(EKS Add-on으로 설치/관리)
Pod Identity는 trust 정책이 cluster·account에 결합되지 않아 운영 부담이 줄어든다 — 같은 Role을 멀티 클러스터·멀티 계정에 그대로 재사용.
