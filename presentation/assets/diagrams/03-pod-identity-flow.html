Lane 1 · Pod
Pod (애플리케이션 + AWS SDK)
namespace: app · ServiceAccount: sa-app
1
Pod 시작
Pod Identity Association이 걸린 SA로 Pod가 스케줄·기동
2
EKS가 env + projected token 주입
AWS_CONTAINER_CREDENTIALS_FULL_URI = http://169.254.170.23/v1/credentials
AWS_CONTAINER_AUTHORIZATION_TOKEN_FILE = /var/run/secrets/pods.eks.amazonaws.com/serviceaccount/eks-pod-identity-token
audience: pods.eks.amazonaws.com · TTL 24h
3
SDK가 Container Credential Provider 진입
env URI 발견 → loopback으로 GET /v1/credentials, Authorization: <token> 헤더 부착
7
Pod가 AWS API 호출
받은 임시 credential로 S3 / DynamoDB / SQS 등 호출 — SDK가 만료 전 자동 갱신
▶
Lane 2 · Agent
Pod Identity Agent (DaemonSet)
hostNetwork · 169.254.170.23 :80
4
노드 loopback에서 token 수신·검증
Pod은 IMDS(169.254.169.254)를 거치지 않는다 — 노드 내 link-local로만 통신.
Agent가 projected JWT를 풀어 audience·exp·subject 확인.
5
EKS Auth API AssumeRoleForPodIdentity 호출
host: eks-auth.<region>.api.aws
auth: 노드 IAM Role (권한: eks-auth:AssumeRoleForPodIdentity)
body: {"token": "<projected JWT>"}
!
STS API 직접 호출 없음
Pod도 Agent도 sts.amazonaws.com을 부르지 않는다.
Token 교환은 EKS Auth API가 처리 → customer STS quota 미사용.
▶
Lane 3 · EKS Auth API
EKS Auth API
eks-auth.<region>.api.aws
6
임시 credential 반환 (SigV4)
accessKeyId · secretAccessKey · sessionToken · expiration
자동 session tag 6종을 transitive로 첨부:
eks-cluster-name, eks-cluster-arn,
kubernetes-namespace, kubernetes-service-account,
kubernetes-pod-name, kubernetes-pod-uid
→ ABAC 조건(aws:PrincipalTag/...)으로 활용 가능
★
Trust policy: 단일 Service Principal
Principal: { "Service": "pods.eks.amazonaws.com" }
OIDC URL·Account ID에 결합되지 않음 → 모든 클러스터·계정에 동일 trust policy 재사용
